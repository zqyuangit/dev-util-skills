# Formal Software Demo PPT Structure

Use this reference when designing a formal software demonstration, product introduction, defense, inspection, or acceptance deck.

## Default 14-Page Structure

Use this structure when the user asks for a formal version around 12-15 pages.

1. **Cover**
   - Software name, project/category, version/status if known
   - One real UI screenshot or product signal
   - No long background paragraphs

2. **Agenda / Evaluation Logic**
   - Show how the deck answers reviewer concerns
   - Example: value, requirement coverage, real UI evidence, engineering delivery

3. **Product Positioning**
   - Target users, usage scenario, core value
   - Use a concise workflow or value loop

4. **Functional Requirement Overview**
   - Completion table or progress chart
   - Include honest pending items

5. **Submission / Acceptance Requirement Mapping**
   - Required artifacts and current status
   - Evidence or next action for each item

6. **Core Feature 1: User/Profile/Data Model**
   - Real UI screenshot
   - 2-3 evaluation points

7. **Core Feature 2: Resource/Content/Business Output**
   - Real UI screenshot(s)
   - Explain what output the software produces

8. **Core Feature 3: Workflow/Path/Process**
   - Real UI screenshot or flow
   - Show how users move from input to result

9. **Core Feature 4: Interaction/Feedback/Operation**
   - Real UI screenshot
   - Show operation entry, user action, and result

10. **Data / Knowledge / Content Credibility**
   - Data source, rule engine, knowledge base, retrieval, or validation flow
   - Avoid pretending unverifiable data exists

11. **Non-Functional Requirements**
   - Runability, maintainability, scalability, privacy/security, usability
   - Use measured or evidence-backed wording

12. **Engineering Delivery**
   - Package structure, runtime components, deployment files, API/backend evidence
   - Show real paths or architecture only when verified

13. **Live Demo Route**
   - Step-by-step sequence for inspection
   - Each step should map to an actual UI page

14. **Closing / Q&A**
   - Concise conclusion
   - Pending materials or next improvements

## Slide Design Rules

- One slide should have one main claim.
- Every non-cover slide should include at least one proof object: real UI, table, diagram, progress chart, package evidence, or workflow.
- Use dense but readable layouts for software evaluation; avoid marketing hero pages except the cover.
- Avoid empty cards with generic nouns. Each card must contain a concrete status, function, action, or evidence.
- Use screenshots as evidence, not decoration.

## Common Page Archetypes

### Requirement Mapping Table

Columns:

- Requirement / artifact
- Status
- Evidence
- Pending work, if any

### Feature Evidence Page

Layout:

- Large real UI screenshot
- Right-side evaluation panel
- 2-4 points: implemented behavior, input/output, review focus, limitation

### Engineering Delivery Page

Layout:

- Left: package/source composition
- Right: validation route or runtime architecture
- Include concrete directory names only when inspected

### Demo Route Page

Layout:

- Real UI screenshot
- Numbered demo steps
- Each step maps to a real page or command
